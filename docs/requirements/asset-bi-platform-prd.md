# 资产分析 BI 平台需求文档

## 1. 项目背景

当前资产数据主要通过 Excel 进行记录、统计和汇总，虽然能满足基础登记需求，但在以下方面存在明显不足：

- 数据展示不直观，难以面向领导层进行汇报
- 统计维度有限，无法灵活按年份、资产类型、部门、地点、供应商等维度分析
- 发货流水、采购记录、资产台账之间缺乏统一关联
- Excel 手工汇总效率低，容易出现口径不一致和数据遗漏
- 现成开源 BI 工具在界面风格、交互体验和业务适配上不满足需求

本项目拟建设一个面向内部使用的资产分析 BI 平台，用于统一管理资产数据、采购数据、发货流水数据，并提供高质量的可视化仪表盘、统计分析页面和可筛选的数据表格，最终用于内部运营管理与领导汇报展示。

## 2. 项目目标

### 2.1 总体目标

建设一套具备资产分析、数据汇总、图表展示、明细查询、导入导出能力的 Web 平台，替代 Excel 的核心统计展示场景，并形成可持续迭代的内部数据分析系统。

### 2.2 目标效果

- 统一资产、采购、发货等核心数据
- 支持多资产类型管理与统计分析
- 支持按时间、地点、部门、供应商等维度进行可视化分析
- 提供适合领导查看的仪表盘式展示界面
- 支持导入历史 Excel 数据并逐步沉淀为标准化数据资产
- 后续可部署到自有云服务器，并通过 GitHub 管理代码版本

## 3. 用户角色

### 3.1 领导层

- 查看核心统计看板
- 关注年度采购、资产总量、分布情况、发货去向、汇总结果
- 需要界面直观、清晰、美观，适合汇报展示

### 3.2 资产管理员

- 维护资产台账
- 导入采购与发货流水
- 查询、筛选、纠错、导出数据

### 3.3 财务或采购相关人员

- 查看采购金额、采购数量、供应商维度汇总
- 核对采购明细与资产入库信息

### 3.4 运维或业务支持人员

- 查询设备去向
- 查看资产状态、发货记录和历史流转情况

## 4. 产品定位

本系统不是通用 BI 平台，而是以资产分析为核心的业务型 BI 系统。

系统目标是：

- 兼顾业务数据管理和数据分析展示
- 提供接近 Grafana 风格的仪表盘体验
- 支持深度定制样式、交互和业务逻辑
- 以自研方式逐步沉淀适配本业务的数据模型和指标体系

## 5. 核心业务范围

### 5.1 资产范围

系统需支持多种资产类型，例如但不限于：

- 服务器
- 存储设备
- 网络设备
- 办公电脑
- 显示器
- 其他 IT 或固定资产

### 5.2 业务范围

系统首期覆盖以下业务数据：

- 资产主数据
- 采购单及采购明细
- 发货单及发货明细
- 资产状态与去向
- 部门、地点、供应商等基础维表

## 6. 主要功能需求

### 6.1 仪表盘总览

首页提供面向领导层和管理层的总览 Dashboard，支持以下能力：

- 核心 KPI 卡片展示
- 年度采购数量与金额趋势图
- 各类资产占比图
- 资产地域或地点分布图
- 发货流向统计图
- 供应商采购汇总图
- 可按时间范围、资产类型、部门、地点进行筛选

建议首页具备较强视觉表现力，整体风格参考 Grafana 的专业仪表盘，但更偏业务汇报场景。

### 6.2 资产台账管理

支持资产主数据的管理与查询：

- 新增、编辑、删除资产
- 支持资产编号、名称、类型、型号、采购日期、价格、供应商等字段
- 支持资产状态管理，例如在库、已发货、在用、维修中、报废
- 支持根据资产类型设置扩展字段
- 支持批量导入和批量导出
- 支持高级筛选、排序、分页、字段显隐

### 6.3 采购管理与分析

支持采购数据维护与统计：

- 录入采购单
- 录入采购明细
- 关联供应商信息
- 支持按年份、月份、资产类型、部门、供应商进行汇总
- 查看采购数量趋势与金额趋势
- 查看不同资产类别采购结构

典型分析问题包括：

- 哪一年采购了多少服务器
- 某一时间段采购金额是多少
- 哪个供应商采购占比最高

### 6.4 发货流水管理与分析

支持发货单、发货明细和流向分析：

- 录入或导入发货单
- 记录发货时间、发货地点、接收地点、接收部门、负责人等信息
- 记录发货资产清单
- 支持查看每笔流水发货到哪里
- 支持按地点、部门、时间、资产类型进行汇总
- 支持查询单个资产的历史流转记录

典型分析问题包括：

- 某时间段发货总量是多少
- 发货主要流向哪些地区或部门
- 某类资产发货占比如何

### 6.5 多维分析

系统需支持统一的筛选器与指标分析能力：

- 时间维度：年、季度、月、周、自定义区间
- 组织维度：部门、业务线
- 地理维度：区域、城市、地点
- 资产维度：资产类型、品牌、型号、状态
- 采购维度：供应商、采购批次
- 发货维度：发货单号、发货目的地、接收方

### 6.6 数据表格展示

系统需提供可视化之外的强表格能力：

- 支持大数据量表格展示
- 支持列排序、列筛选、关键字搜索
- 支持多条件组合筛选
- 支持分组汇总
- 支持冻结列、列宽调整、列显隐
- 支持导出 Excel

### 6.7 Excel 数据导入

考虑现阶段大量历史数据仍在 Excel 中，系统首期必须具备导入能力：

- 上传 Excel 文件
- 自动识别表头并生成字段建议
- 支持字段映射配置
- 导入校验与错误提示
- 支持预览导入结果
- 支持导入日志记录
- 支持导入失败原因下载
- 支持“原样导入”模式，避免因字段未预设导致无法入库
- 支持按上传表结构自动创建业务字段或扩展字段
- 支持保存导入模板，便于同类 Excel 后续重复导入
- 支持同一类数据源的字段增量同步，例如下次上传时新增列可自动识别

建议采用“原始导入区 + 标准化层 + 分析层”的处理方式，确保 Excel 即使字段不固定也可以先入库，再逐步清洗和映射。

导入设计原则：

- 不能因为系统未预设某个字段就拒绝导入整份 Excel
- 系统必须优先保证“先接住数据”，再做字段映射、数据清洗和统计建模
- 对无法识别的数据列，系统应自动生成扩展字段并保留原始值
- 对导入失败的数据，必须明确提示失败原因和出错行号
- 原始导入数据需可追溯，方便后续重新映射和重新入库

### 6.8 权限管理

系统需支持基础权限控制：

- 登录认证
- 角色管理
- 菜单权限控制
- 数据查看权限控制

首期角色建议包括：

- 超级管理员
- 领导查看角色
- 资产管理员
- 普通查看角色

### 6.9 导出与汇报支持

为了满足汇报场景，系统应支持：

- 导出 Excel
- 导出 PDF
- 仪表盘截图或打印优化
- 关键指标卡片适配大屏或汇报屏展示

## 7. 非功能需求

### 7.1 界面体验

- 风格需专业、现代、简洁，参考 Grafana 的仪表盘质感
- 不采用过于朴素的传统后台风格
- 深色或中性科技风主题优先，同时保留后续支持浅色主题的能力
- 图表、卡片、表格、筛选器在视觉风格上保持统一

### 7.2 性能要求

- 常规列表查询应在 3 秒内返回
- 常规汇总分析应在 5 秒内返回
- 支持至少数万级明细数据的日常查询和统计
- 高频聚合查询可通过缓存或物化视图优化

### 7.3 可维护性

- 前后端分层清晰
- 数据模型可扩展，支持新增资产类型
- 指标口径统一，避免重复统计逻辑散落在前端
- 代码结构适合后续通过 AI 协助持续开发

### 7.4 安全性

- 支持登录鉴权
- 关键操作保留日志
- 导入功能需校验文件格式与数据合法性
- 数据库连接和环境变量需安全管理

## 9. 系统架构建议

建议采用前后端分离但统一仓库管理的方式：

- 前端：负责页面、图表、表格、交互
- 后端：负责鉴权、业务逻辑、数据查询、导入任务
- 数据库：负责业务数据与分析视图
- 缓存与任务：负责导入任务、缓存热点统计结果

建议采用 Monorepo 结构，便于统一管理：

- `apps/web`：前端项目
- `apps/api`：后端项目
- `packages/ui`：共享 UI 组件
- `packages/config`：共享配置
- `docs`：项目文档

## 10. 数据模型建议

数据模型不建议采用“全部固定字段”的传统方式，而应采用“固定系统字段 + 动态业务字段 + 原始数据留存”的半结构化模型。

### 10.1 设计原则

- 系统级字段固定，例如主键、数据源、创建时间、更新时间、导入批次、状态
- 业务字段支持动态扩展，避免因为 Excel 新增列而必须改表结构
- 原始上传数据完整保留，确保任何 Excel 都可以先接入
- 常用分析字段逐步沉淀为标准字段，用于高性能统计
- 动态字段定义、字段映射规则、导入模板必须独立管理

### 10.2 核心表建议

首期建议至少包含以下核心表：

- `asset_types`：资产类型
- `assets`：资产主表
- `departments`：部门表
- `locations`：地点表
- `vendors`：供应商表
- `purchase_orders`：采购单
- `purchase_order_items`：采购明细
- `shipments`：发货单
- `shipment_items`：发货明细
- `asset_status_history`：资产状态流转历史
- `users`：用户表
- `roles`：角色表
- `import_jobs`：导入任务
- `import_logs`：导入日志

为支持动态字段和自适应导入，建议补充以下能力表：

- `data_sources`：数据源定义，例如资产导入、采购导入、发货导入
- `field_definitions`：字段定义表，记录字段名、显示名、类型、是否必填、是否可筛选、是否可统计
- `field_mappings`：导入映射规则，记录 Excel 列名到系统字段的映射关系
- `raw_import_records`：原始导入记录，完整保存 Excel 原始行数据
- `dynamic_records` 或业务表中的 `jsonb` 扩展字段：保存未标准化或动态扩展的数据列
- `import_templates`：导入模板定义，支持同类 Excel 重复使用

### 10.3 推荐存储策略

推荐使用以下组合模式，而不是单纯依赖固定列：

- 业务主表保留少量固定核心列
- 业务扩展字段统一存入 `jsonb`
- 原始 Excel 行数据完整保存到 `raw_import_records`
- 字段定义存储在 `field_definitions`
- Excel 列与系统字段的映射关系存储在 `field_mappings`

以 `assets` 为例，建议结构如下：

- 固定列：`id`、`source_id`、`import_job_id`、`asset_type_id`、`status`、`created_at`、`updated_at`
- 标准业务列：后续高频分析字段，如采购日期、采购金额、地点、部门，可逐步沉淀
- 扩展列：`extra_data jsonb`，用于保存当前 Excel 中的非标准字段

### 10.4 关于“完全不设固定字段”的说明

从导入兼容性角度看，系统应尽量减少固定业务字段约束；但从系统实现角度看，仍然必须保留少量固定系统字段，否则会出现以下问题：

- 无法建立主键与数据关系
- 无法做权限控制、审计日志和导入追踪
- 无法进行标准化汇总和高性能统计
- 后续前端筛选、图表分析和导出逻辑会失去统一基础

因此推荐方案不是“零固定字段”，而是：

- 固定系统字段最小化
- 业务字段动态化
- 原始数据全量保留
- 高频分析字段逐步标准化

### 10.5 导入与建模流程建议

建议每次 Excel 导入按以下流程处理：

1. 上传 Excel，识别 Sheet、表头、数据类型
2. 将原始行完整写入 `raw_import_records`
3. 对已识别字段按映射规则写入标准业务表
4. 对未识别字段自动写入 `jsonb` 扩展字段，并生成字段定义
5. 对关键统计字段允许人工确认是否提升为正式标准字段
6. 保存本次导入模板，供下次相同格式复用

### 10.6 统计分析策略

为了兼顾“Excel 任意导入”和“BI 高性能分析”，建议分层处理：

- 原始层：保存完整 Excel 数据，不做强约束
- 标准层：沉淀清洗后的业务字段，支持列表查询和表单维护
- 分析层：基于标准层生成统计视图、物化视图和仪表盘指标

这样既能兼容用户上传的不同表格，又不会让后续 BI 查询完全依赖 `jsonb`，避免统计性能和可维护性失控。

扩展设计原则：

- 优先允许导入，避免字段不一致导致整表失败
- 通用高频字段逐步结构化建模
- 动态字段通过 `jsonb + 字段定义表` 管理
- 统计高频字段必须单独建列，避免后续查询性能下降
- 原始导入记录必须长期保留，确保可追溯与可重建

## 15. 后续开发原则

- 先完成业务闭环，再逐步增强通用 BI 能力
- 先统一数据口径，再扩展图表数量
- 优先保证领导汇报场景的可用性和美观性
- 所有统计指标应由后端统一输出，避免前端重复计算
